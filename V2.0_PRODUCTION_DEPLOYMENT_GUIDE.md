# V2.0 Production Deployment Guide

## Overview
This guide provides step-by-step instructions for deploying the V2.0 arbitrage strategy system to production.

## Pre-Deployment Requirements

### 1. System Requirements
- **OS**: Windows Server 2022 or Linux (Ubuntu 20.04+)
- **Python**: 3.9+
- **Node.js**: 16+
- **Database**: PostgreSQL 13+
- **Memory**: 8GB+ RAM
- **CPU**: 4+ cores
- **Disk**: 100GB+ SSD

### 2. Software Dependencies
- Git
- Python pip
- Node.js npm
- PostgreSQL client
- Redis (for caching)
- Nginx (for reverse proxy)

### 3. Exchange Accounts
- Bybit account with API keys
- MT5 account with credentials
- Sufficient balance for trading

### 4. Network Requirements
- Stable internet connection
- Low latency to exchange APIs (<100ms)
- Firewall rules configured
- SSL certificates installed

## Deployment Steps

### Step 1: Backup Current System

```bash
# 1. Backup database
pg_dump -h localhost -U hustle_user -d hustle_db > backup_$(date +%Y%m%d_%H%M%S).sql

# 2. Backup configuration files
tar -czf config_backup_$(date +%Y%m%d_%H%M%S).tar.gz backend/config/

# 3. Tag current version
git tag -a v1.0-pre-v2 -m "Pre-V2.0 deployment backup"
git push origin v1.0-pre-v2

# 4. Create system snapshot (if using VM)
# Follow your cloud provider's snapshot procedure
```

### Step 2: Stop Current Services

```bash
# Stop backend service
systemctl stop hustle-backend

# Stop frontend service (if separate)
systemctl stop hustle-frontend

# Verify services stopped
systemctl status hustle-backend
systemctl status hustle-frontend
```

### Step 3: Update Code

```bash
# Navigate to project directory
cd /app/hustle2026

# Fetch latest code
git fetch origin

# Checkout V2.0 release branch
git checkout v2.0-release

# Verify correct version
git log -1
```

### Step 4: Update Dependencies

```bash
# Update Python dependencies
cd backend
pip install -r requirements.txt --upgrade

# Update Node.js dependencies
cd ../frontend
npm install
npm audit fix
```

### Step 5: Database Migration

```bash
# Navigate to backend directory
cd /app/hustle2026/backend

# Review pending migrations
alembic current
alembic history

# Run migrations
alembic upgrade head

# Verify migration success
alembic current
```

### Step 6: Update Configuration

```bash
# Update production configuration
cd /app/hustle2026/backend/config

# Edit production.yaml
nano production.yaml
```

**Key Configuration Updates**:
```yaml
# V2.0 Strategy Configuration
strategy:
  v2:
    enabled: true
    trigger_timeout: 0.2  # seconds
    order_timeout: 0.1    # seconds
    max_retries: 3

# WebSocket Configuration
websocket:
  enabled: true
  heartbeat_interval: 30
  max_connections: 100

# Exchange Configuration
exchanges:
  bybit:
    api_key: ${BYBIT_API_KEY}
    api_secret: ${BYBIT_API_SECRET}
    testnet: false
  mt5:
    server: ${MT5_SERVER}
    login: ${MT5_LOGIN}
    password: ${MT5_PASSWORD}
```

### Step 7: Build Frontend

```bash
# Navigate to frontend directory
cd /app/hustle2026/frontend

# Build production bundle
npm run build

# Verify build output
ls -lh dist/
```

### Step 8: Deploy Frontend

```bash
# Copy build to web server
cp -r dist/* /var/www/hustle/

# Set permissions
chown -R www-data:www-data /var/www/hustle/
chmod -R 755 /var/www/hustle/

# Restart web server
systemctl restart nginx
```

### Step 9: Start Backend Services

```bash
# Start backend service
systemctl start hustle-backend

# Check service status
systemctl status hustle-backend

# Monitor logs
tail -f /var/log/hustle/backend.log
```

### Step 10: Verify Deployment

```bash
# Run smoke tests
cd /app/hustle2026/backend
python -m pytest tests/smoke_tests.py -v

# Check API health
curl http://localhost:8000/api/v1/health

# Check WebSocket connection
wscat -c ws://localhost:8000/ws
```

## Rollback Procedure

If deployment fails or critical issues are discovered:

### Step 1: Stop Services

```bash
systemctl stop hustle-backend
systemctl stop hustle-frontend
```

### Step 2: Restore Code

```bash
cd /app/hustle2026
git checkout v1.0-pre-v2
```

### Step 3: Restore Database

```bash
# Stop backend to prevent connections
systemctl stop hustle-backend

# Restore database
psql -h localhost -U hustle_user -d hustle_db < backup_YYYYMMDD_HHMMSS.sql

# Verify restoration
psql -h localhost -U hustle_user -d hustle_db -c "SELECT version FROM alembic_version;"
```

### Step 4: Restore Configuration

```bash
cd /app/hustle2026/backend/config
tar -xzf config_backup_YYYYMMDD_HHMMSS.tar.gz
```

### Step 5: Restart Services

```bash
systemctl start hustle-backend
systemctl start hustle-frontend
```

### Step 6: Verify Rollback

```bash
# Check service status
systemctl status hustle-backend

# Run smoke tests
python -m pytest tests/smoke_tests.py -v

# Check logs
tail -f /var/log/hustle/backend.log
```

## Gradual Rollout Strategy

For safer deployment, consider gradual rollout:

### Phase 1: Canary Deployment (10% traffic)
1. Deploy V2.0 to canary server
2. Route 10% of traffic to canary
3. Monitor for 24 hours
4. If stable, proceed to Phase 2

### Phase 2: Partial Rollout (50% traffic)
1. Deploy V2.0 to half of production servers
2. Route 50% of traffic to V2.0
3. Monitor for 48 hours
4. If stable, proceed to Phase 3

### Phase 3: Full Rollout (100% traffic)
1. Deploy V2.0 to all production servers
2. Route 100% of traffic to V2.0
3. Monitor for 1 week
4. Decommission V1.0 servers

## Performance Optimization

### 1. Database Optimization

```sql
-- Create indexes for V2.0 queries
CREATE INDEX idx_strategy_status ON strategies(status);
CREATE INDEX idx_trigger_timestamp ON triggers(timestamp);
CREATE INDEX idx_position_strategy ON positions(strategy_id);

-- Analyze tables
ANALYZE strategies;
ANALYZE triggers;
ANALYZE positions;
```

### 2. Application Optimization

```python
# Enable connection pooling
SQLALCHEMY_POOL_SIZE = 20
SQLALCHEMY_MAX_OVERFLOW = 40

# Enable query caching
REDIS_CACHE_TTL = 300

# Enable async processing
ASYNC_WORKERS = 4
```

### 3. Web Server Optimization

```nginx
# Nginx configuration
worker_processes auto;
worker_connections 1024;

# Enable gzip compression
gzip on;
gzip_types text/plain application/json;

# Enable caching
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=my_cache:10m;
```

## Security Considerations

### 1. API Security
- Use HTTPS for all API endpoints
- Implement rate limiting
- Validate all input data
- Use JWT tokens with expiration
- Implement CORS properly

### 2. Database Security
- Use strong passwords
- Limit database user permissions
- Enable SSL connections
- Regular security updates
- Backup encryption

### 3. Exchange API Security
- Store API keys in environment variables
- Use IP whitelisting
- Enable 2FA on exchange accounts
- Monitor API usage
- Rotate keys regularly
