# 套利策略对比分析报告

## 执行摘要

**结论：当前系统已实现新策略的核心逻辑，仅需微调Binance超时时间（0.4秒→0.3秒）即可完全符合新策略要求。**

## 详细对比分析

### 1. 超时时间设置

| 项目 | 当前系统 | 新策略要求 | 是否需要修改 |
|------|---------|-----------|-------------|
| Binance超时 | 0.4秒 (400ms) | 0.3秒 (300ms) | ✅ **需要修改** |
| Bybit超时 | 0.1秒 (100ms) | 0.1秒 (100ms) | ✅ 无需修改 |

**修改位置**：`backend/app/services/order_executor_v2.py` 第20行
```python
# 当前
self.binance_timeout = 0.4  # 400ms

# 修改为
self.binance_timeout = 0.3  # 300ms
```

---

### 2. Binance订单监控逻辑

| 项目 | 当前系统 | 新策略要求 | 评估 |
|------|---------|-----------|------|
| 监控方式 | 持续检测（每10ms检测一次）直到超时 | 等待0.3秒后检测一次 | ✅ **当前实现更优** |
| 超时处理 | 超时后撤单并返回已成交数量 | 0.3秒后检测，未成交则撤单 | ✅ 逻辑相同 |
| 部分成交 | 返回部分成交数量 | 撤单并返回部分成交数量 | ✅ 逻辑相同 |

**建议**：保持当前的持续检测逻辑，因为：
- 更实时（能在0.3秒内任何时刻检测到成交）
- 更可靠（不会错过成交时机）
- 结果相同（0.3秒后都会撤单未成交部分）

    # 未完全成交则撤单并继续循环

### 6. 反向开仓策略对比

#### 当前系统实现
```
1. Binance ask价挂单开空（LIMIT + POST_ONLY）
2. 持续检测0.4秒，超时撤单
3. Bybit按Binance成交数量ask价吃单开多（Market）
4. 0.1秒后检测，未完全成交则撤单重试1次
```

#### 新策略要求
```
1. Binance ask价挂单开空
2. 0.3秒后检测，未成交或部分成交则撤单
3. Bybit按Binance成交数量ask价吃单开多
4. 0.1秒后检测，未完全成交则撤单重试1次
```

**差异**：仅Binance超时时间（0.4秒 vs 0.3秒）
